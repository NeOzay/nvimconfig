# claudecode.nvim

## Role
Pont entre Neovim et Claude Code (CLI dans un terminal externe) : partage de fichiers et de sélections.

## Files
- Config: `lua/plugins/claudecode/init.lua`
- Provider terminal custom: `lua/plugins/claudecode/terminal.lua`
- Highlights: `lua/highlights/claudecode.lua`

## Key Behaviors
- Usage : envoyer le buffer courant ou une sélection visuelle à Claude Code.
- Terminal provider : **custom** (`lua/plugins/claudecode/terminal.lua`), pas le provider `"snacks"` intégré au plugin. `claudecode.nvim` accepte une table custom comme `terminal.provider` (contrat `setup/open/close/simple_toggle/focus_toggle/get_active_bufnr/is_available`, voir `claudecode/types.lua` du plugin) ; on l'utilise pour posséder notre propre suivi buf/win au lieu de subir celui de `snacks.win` (voir Gotchas historique et section suivante). Claude Code tourne dans un split docké à droite (`<A-c>` pour toggle en mode n et t, `position = "right"` dans `snacks_win_opts`).
- `lazy = false` — chargé immédiatement.
- `<leader>a` réservé comme préfixe "AI/Claude Code".
- `diff_opts.layout = "vertical"` par défaut ; le mode `"unified"` (diff inline dans un seul buffer) utilise les groupes de highlight définis dans `lua/highlights/claudecode.lua` : `ClaudeCodeInlineDiffAdd`, `ClaudeCodeInlineDiffDelete`, `ClaudeCodeInlineDiffAddSign`, `ClaudeCodeInlineDiffDeleteSign`. Bg add/delete réutilisent les couleurs `CodeDiffLineInsert`/`CodeDiffLineDelete` (base46/config.lua) pour rester cohérent avec codediff.nvim ; les signs utilisent `colors.green`/`colors.red`.
- `opts.env = { EDITOR = "nvim" }` — fixe `$EDITOR` uniquement dans le terminal Claude Code (pas globalement). Combiné à [[nvim-unception]], Ctrl+G dans Claude Code (édition du prompt) ouvre le fichier dans l'instance Nvim hôte au lieu d'imbriquer un Nvim dans le terminal.
- `diff_opts.hide_terminal_in_new_tab = false` — le terminal reste visible dans l'onglet diff (layout source | diff | terminal).

## Provider terminal custom (`lua/plugins/claudecode/terminal.lua`)
Remplace le provider `"snacks"` intégré au plugin tiers. Ce dernier expose `get_active_bufnr()` en lisant `terminal.buf`, un champ interne de `snacks.win` que `fixbuf` peut désynchroniser (voir Gotchas historique ci-dessous) — notre provider évite ce piège en ne dépendant jamais de ce champ pour son API publique :
- `term_buf` : référence stable au vrai buffer terminal, capturée une seule fois à la création, jamais réassignée. Seule source de vérité pour `get_active_bufnr()`.
- `guard_window()` : quand un buffer étranger remplace le terminal dans sa fenêtre (navigation cokeline, `gf`, tout code qui réutilise la fenêtre courante), on délègue à `terminal:fixbuf()` — la méthode native de `snacks.win`, déjà responsable de ce comportement pour ses propres terminaux — plutôt que de réimplémenter l'éjection à la main ; deux approches maison essayées et abandonnées avant, voir Gotchas historique. `terminal.opts.fixbuf` (forcé à `false` à l'ouverture, voir ci-dessous) est basculé à `true` juste le temps de l'appel, uniquement si le buffer courant de la fenêtre est étranger (ni `term_buf`, ni le prompt Ctrl+G), puis repassé à `false` — pour que `fixbuf()` reste un no-op sur `term_buf` et sur le prompt, qui doivent pouvoir s'afficher en place. Un garde `nvim_get_current_win() == terminal.win` filtre les `BufWinEnter` qui ne concernent pas notre fenêtre. L'autocmd `BufWinEnter` interne que `snacks.win` recrée à chaque `show()` est explicitement supprimée (`setup_terminal_events()`) pour éviter que les deux handlers ne se marchent dessus — c'était la cause du bug winhighlight constaté le 2026-07-24 (voir Changelog).
- `last_win_size` : taille (largeur/hauteur en cellules) capturée juste avant `hide()`, réappliquée sur `terminal.opts.width/height` avant le `show()` suivant — `snacks.win` relit ces champs à chaque `show()` (`M:dim()`), donc le terminal garde sa taille manuelle d'un toggle à l'autre au lieu de revenir aux pourcentages par défaut.
- L'autocmd `QuitPre` (pattern `*claude-prompt-*.md`) fait aussi le ménage du buffer prompt lui-même : `nvim-unception` ne le wipe jamais après le `:wq` (juste `split`/`buffer <term_buf>`/`wincmd x` puis fermeture de fenêtre), et le CLI Claude supprime le fichier tmp dès qu'il l'a lu — un buffer chargé pointant sur un fichier disparu se fait piéger par le balayage de timestamps automatique de Neovim (retour de focus après le job terminal) et lève `E211`. On `nvim_buf_delete` le buffer prompt en différé, seulement si plus aucune fenêtre ne l'affiche (quit effectivement abouti, pas annulé pour modifs non sauvegardées). `startinsert()` est aussi rappelé à la fin de ce handler pour repasser en insert dans le terminal au retour du Ctrl+G.
- Auto-insert à l'ouverture du prompt Ctrl+G : dans le handler `BufWinEnter`, quand `is_prompt_buf(buf)` est vrai, `vim.defer_fn` (50 ms) place le curseur en fin de buffer (`normal! G`) puis `startinsert!`. Passe par l'API (`nvim_win_call` + `vim.cmd`) plutôt que `nvim_feedkeys("GA")` : ce handler est `nested = true` et peut se redéclencher plusieurs fois pour le même buffer (nvim-unception fait `0argadd` + `argument` + `edit`), chaque déclenchement empilant ses touches dans le typeahead — au second passage, `G`/`A` étaient tapées alors qu'on était déjà en insert (entré au premier passage) et s'inséraient littéralement comme texte (`"GA"` visible dans le buffer). `normal!`/`startinsert!` sont idempotents : les rappeler plusieurs fois ne réinsère rien.
- `M.reset_size()` : efface `last_win_size` et réapplique `snacks_win_opts.width/height` (capturés dans `default_win_size` à chaque ouverture) sur `terminal.opts`. Si la fenêtre est visible, redessine immédiatement via `vertical resize`/`resize` (selon `position` gauche/droite ou haut/bas) plutôt que d'attendre le prochain `show()` — `snacks.win` n'expose pas d'API de redimensionnement à chaud pour un split déjà ouvert, seule `open_win()` applique la taille (à la création). Exposé via `<leader>aR`.
- `snacks_win_opts.fixbuf = false` est forcé en interne par le provider (pas un réglage exposé dans `init.lua`) — indispensable pour que le prompt Ctrl+G puisse s'afficher en place sans que le `fixbuf` par défaut de snacks le swap immédiatement hors de la fenêtre.
- Réutilise `claudecode.utils` (`normalize_focus`, `parse_command`) du plugin tiers — module public, pas de duplication de la logique de parsing de commande. Requis en lazy (à l'intérieur des fonctions, pas en haut de fichier) car `claudecode.nvim` n'est pas encore sur le runtimepath au moment où lazy.nvim évalue notre spec.

## Keymaps
| Touche | Mode | Action |
|--------|------|--------|
| `<leader>am` | n | Sélectionner le modèle Claude |
| `<leader>ab` | n | Ajouter le buffer courant au contexte |
| `<leader>as` | v | Envoyer la sélection à Claude |
| `<leader>as` | n (ft: explorer) | Ajouter fichier (ClaudeCodeTreeAdd) |
| `<leader>aa` | n | Accepter diff |
| `<leader>ad` | n | Refuser diff |
| `<leader>aR` | n | Réinitialiser la taille du terminal aux valeurs par défaut |

## Gotchas
- Les commandes de toggle terminal (`ClaudeCode`, `ClaudeCodeFocus`, `--resume`, `--continue`) sont commentées — non utilisées.
- **Historique résolu (2026-07-23)** : avant le provider custom, `snacks_win_opts.fixbuf = false` (nécessaire pour Ctrl+G) corrompait de façon permanente le suivi interne du terminal par `snacks.win` (`self.buf` restait bloqué sur le buffer du prompt après édition), avec un filet de sécurité dans `lua/autocmds.lua` (cache `TermOpen` + réparation sur `QuitPre` via une fonction privée du plugin tiers, `claudecode.terminal.snacks._get_terminal_for_test()`). Le provider custom élimine ce problème à la racine : voir section précédente. Le hack et son filet de sécurité ont été supprimés.
- Le contournement "climbing cursor" du provider `snacks` de référence (`coder/claudecode.nvim`, destroy+recreate de fenêtre qui décale le curseur Ink d'une ligne à chaque toggle) n'est **pas** porté dans notre provider — non constaté chez nous, à ajouter seulement si observé.
- **Trois approches de garde de fenêtre essayées et abandonnées avant la version actuelle (délégation à `terminal:fixbuf()`, voir section précédente)** :
  1. Éjection réactive maison du buffer étranger vers une fenêtre "principale" de l'onglet (`nvim_win_set_buf` + `sbuffer` différé) : faisait fuiter les options de fenêtre du terminal (`winhighlight`, `statuscolumn`) vers la fenêtre/le buffer relogés. Cause exacte non élucidée avec certitude à l'époque (candidats envisagés : `Snacks.util.wo`/`update()` réappliqué sur la mauvaise fenêtre à cause de `terminal.buf` corrompu par `fixbuf = false`, héritage d'options au split, `stack = true` du style terminal snacks) — la cause réelle, confirmée le 2026-07-24, était le second autocmd `BufWinEnter` que `snacks.win` recrée en interne à chaque `show()`, qui entrait en conflit avec `guard_window()` (les deux géraient le même événement) ; supprimé explicitement dans `setup_terminal_events()`.
  2. Verrou natif `'winfixbuf'` (Neovim 0.10+) : empêche structurellement tout remplacement de buffer dans la fenêtre terminal (`E1513`), levé ponctuellement pendant Ctrl+G via l'autocmd `User UnceptionEditRequestReceived` de nvim-unception. Fonctionnait pour Ctrl+G, mais `nvim_set_current_buf`/`:buffer` échouent alors *pour tout code*, pas seulement les intrusions non désirées — cokeline (navigation par onglets) n'enveloppe pas ses appels dans `pcall` et plantait avec `E1513` dès que le focus était dans le terminal. Trop strict pour un environnement où d'autres plugins ne s'attendent pas à ce qu'une fenêtre refuse un changement de buffer.
  3. "Laisser faire" : le buffer étranger remplace le terminal normalement, `terminal.buf` réparé immédiatement, terminal rouvert ailleurs via `show()`. Fonctionnait mais réinventait à la main un comportement déjà présent dans `snacks.win` (`fixbuf()`) — remplacé par une délégation directe à cette méthode, plus la suppression de l'autocmd `BufWinEnter` interne qui était la vraie cause de la fuite `winhighlight`/`statuscolumn` observée depuis le point 1.

## Changelog
- 2026-06-05 : Analyse initiale. Usage clarified : communication uniquement, terminal externe.
- 2026-07-22 : Identifié et corrigé la cause racine du terminal qui reste bloqué sur le buffer du prompt après Ctrl+G : `fixbuf = false` fait adopter le buffer du prompt en permanence par l'objet `snacks.win` (`self.buf`). Fix dans `lua/autocmds.lua` (cache `TermOpen` + réparation via `snacks.win:set_buf()` sur `QuitPre`), `hide_terminal_in_new_tab` reste `false`.
- 2026-07-23 : Remplacé le provider `"snacks"` intégré par un provider custom (`lua/plugins/claudecode/terminal.lua`) qui possède son propre suivi buf/win — élimine le hack `fixbuf`/`QuitPre` à la racine. Ajout de la persistance de taille du terminal entre toggles et de la garde de fenêtre (seuls le terminal et le prompt Ctrl+G peuvent occuper cette fenêtre). Conversion `lua/plugins/claudecode.lua` → `lua/plugins/claudecode/init.lua` (convention dossier du repo).
- 2026-07-23 : Remplacé la garde de fenêtre par éjection réactive (`BufWinEnter` + relogement vers une fenêtre "principale") par un verrou natif `'winfixbuf'` — l'éjection faisait fuiter les options de fenêtre du terminal (`winhighlight`, `statuscolumn`) vers le buffer relogé, cause exacte non élucidée. `winfixbuf` empêche structurellement tout remplacement de buffer dans la fenêtre terminal, levé ponctuellement pendant l'édition Ctrl+G via l'autocmd `User UnceptionEditRequestReceived`.
- 2026-07-23 : `'winfixbuf'` abandonné à son tour — cassait `nvim_set_current_buf`/`:buffer` pour tout code tiers non défensif (cokeline plantait avec `E1513` lors d'un clic sur un onglet). Remplacé par une approche "laisser faire" : le buffer étranger remplace le terminal normalement, `terminal.buf` est réparé immédiatement pour empêcher toute fuite de style, et le terminal est rouvert ailleurs via `show()` (qui recrée une fenêtre fraîche sans jamais toucher à celle abandonnée).
- 2026-07-24 : Erreur `E211: File "..." no longer available` constatée une fois par session à la fermeture du buffer prompt (`:wq` sur `claude-prompt-*.md`) — cause : `nvim-unception` ne wipe jamais ce buffer, et le CLI Claude supprime le fichier tmp dès qu'il l'a lu ; le prochain balayage de timestamps de Neovim tombe dessus. Fix : `nvim_buf_delete` différé du buffer prompt dans l'autocmd `QuitPre`, une fois le quit effectivement abouti.
- 2026-07-24 : Ajout de `M.reset_size()` (keymap `<leader>aR`) — réinitialise la taille du terminal aux valeurs par défaut (`snacks_win_opts.width/height`, capturées dans `default_win_size` à l'ouverture) et redessine la fenêtre à chaud si elle est visible (`vertical resize`/`resize` direct, `snacks.win` n'exposant pas d'API de redimensionnement à chaud pour un split). Type custom `Ozay.ClaudeCodeTerminalProvider` (étend `ClaudeCodeTerminalProvider` du plugin tiers) pour documenter ce champ additionnel.
- 2026-07-24 : Auto-insert ajouté à l'ouverture du prompt Ctrl+G (`normal! G` + `startinsert!` en différé dans `BufWinEnter`) et `startinsert()` au retour dans le terminal (`QuitPre`). Premier essai avec `nvim_feedkeys("GA")` buggé : le handler `BufWinEnter` (`nested = true`) se redéclenche plusieurs fois pour le même buffer, empilant les touches dans le typeahead — au second passage `G`/`A` s'inséraient littéralement comme texte car déjà en insert. Remplacé par des appels API idempotents.
- 2026-07-24 : Bug winhighlight/statuscolumn identifié — `guard_window()` entrait en conflit avec l'autocmd `BufWinEnter` interne que `snacks.win` recrée à chaque `show()` (double gestion du même événement). `setup_terminal_events()` supprime maintenant cet autocmd interne. `guard_window()` réécrit pour déléguer à `terminal:fixbuf()` (méthode native de `snacks.win`) au lieu de la logique "laisser faire + réparer + rouvrir" — pilotée en basculant `terminal.opts.fixbuf` à `true` le temps de l'appel, uniquement pour un buffer étranger. `last_win_size` réinitialisé sur `TermClose`/`BufWipeout` (fuite possible vers une future instance sinon). `lua/plugins/statuscol/init.lua` passé de `event = "VeryLazy"` à `lazy = false` (statuscolumn pas prête au premier `show()` du terminal, contribuait à la confusion).
